# Third Eye Worldwide — Next.js Site

A Next.js 14 (App Router) site for Third Eye Worldwide (TEWW). Each route is a server component that injects HTML generated by a render module under `lib/pages/`; navigation, theme, and text-size controls live in client components.

## Run locally

```bash
npm install
npx prisma migrate dev          # create / migrate SQLite dev DB
npm run db:seed                 # load data/seed.json and create admin user
npm run dev
```

Open `http://localhost:3000` (site) or `http://localhost:3000/admin` (CMS).

**CMS publish token** — set in `.env` as `CMS_TOKEN`. First time you click **Publish** in the dashboard, you'll be prompted to paste the token; it's cached in localStorage thereafter.

## Build

```bash
npm run build
npm start
```

## Deploy

### Vercel
Push to GitHub and import in Vercel — Next.js is auto-detected. `vercel.json` adds long-lived cache headers for fonts/assets and security headers.

### Netlify
Connect the repo; `netlify.toml` requires the `@netlify/plugin-nextjs` plugin (installed automatically by Netlify for Next.js projects).

### Other hosts
Any Node-compatible host that supports Next.js 14 will work.

## CMS & database

- **DB**: Prisma 6 with SQLite for dev (`prisma/dev.db`). Swap to Postgres for prod by editing `datasource db { provider = "postgresql" }` in `prisma/schema.prisma` and setting `DATABASE_URL` to a Postgres connection string (Neon / Supabase / RDS) — no schema changes required.
- **Schema** (`prisma/schema.prisma`):
  - `SiteContent` — single-row JSON document holding all editable content (mirrors `data/seed.json`).
  - `ContentRevision` — append-only history of every publish.
  - `VolunteerSubmission` / `DonationSubmission` — form submissions.
  - `User` — admin accounts (seeded from `CMS_ADMIN_EMAIL` / `CMS_ADMIN_PASSWORD`).
- **Data access**: `lib/cms/db.js` — `getContent()`, `saveContent()`, `visibleSorted()`.
- **API**:
  - `GET /api/cms/data` — public read.
  - `PUT /api/cms/data` — requires `x-cms-token` header; writes a new revision, updates current, and calls `revalidatePath()` on every route.
  - `POST /api/cms/submissions/volunteer` and `/donation` — public writes (for forms).
  - `GET` on submission routes — requires token, returns the last 200 rows.
- **Admin dashboard**: `/admin` (served from `public/admin/`). Loads from `GET /api/cms/data`; **Publish** button does `PUT /api/cms/data`. Show/hide an item via its `visible` flag; reorder via `order`.

### Phases 1, 2 & 3 (done)
- **All 9 content routes fully CMS-driven**: `/`, `/about`, `/projects`, `/donate`, `/media`, `/documents`, `/volunteers`, `/blog-detail`, `/story-detail`. Every item honours `visible` + `order`.
- CMS dashboard at `/admin` edits every collection; Publish → DB → `revalidatePath()` on all 9 fixed routes + any user-defined `/slug` pages.
- Form submissions wired: Donate and Volunteer forms POST to `/api/cms/submissions/{donation,volunteer}` and land in the DB for admin review.
- **Auth (Phase 3)**: NextAuth v4 credentials provider backed by the Prisma `User` table. `/admin/login` signs in, middleware (`middleware.js`) gates every `/admin` route, and `/api/cms/*` writes require a session (legacy `x-cms-token` still accepted for scripted clients via `lib/cms/auth-guard.js`).
- **Visibility + reorder UI (Phase 3)**: every item in the dashboard has a one-click show/hide button (with a striped "hidden" treatment + badge) and Move-up/Move-down buttons that auto-renumber the `order` field.
- **Catch-all dynamic pages (Phase 3)**: `app/[slug]/page.js` renders any page in `content.pages[]`. The Custom pages section in the CMS sidebar lets editors add/remove/hide pages with a free-form HTML body — the slug becomes the live URL. Reserved slugs (`about`, `projects`, `api`, `admin`, etc.) are blocked to prevent shadowing fixed routes.

### Deferred (future)
- Full article body editing for `/blog-detail` / `/story-detail` (header metadata is CMS-driven; body copy is still a hardcoded example article — needs rich-text schema + editor).
- Multi-section composable custom pages (current custom pages use a single `rich` HTML section per page; `lib/pages/custom.js` already supports `hero`/`text`/`rich`/`cta` section types, but the CMS UI exposes only the single-section shape).
- Rich-text (WYSIWYG) editor in place of raw-HTML textareas for long-form fields.

## Features
- 3 themes (light, dark, high-contrast) with localStorage persistence.
- 3 text-size settings.
- 9 routes with sub-nav tabs: Home, About (Mission/Team), Projects, Donate, Media (Photos/Podcasts/Videos), Documents (Blogs/Stories), Volunteers, Blog detail, Story detail.
- Phosphor Icons via CDN; hero photography via Unsplash.
- Skip link, focus rings, ARIA landmarks.

## Notes
- Phosphor Icons and hero imagery load from CDNs (`unpkg.com`, `images.unsplash.com`). For an air-gapped deploy, self-host these assets.
- Donate + Volunteer forms have API endpoints but are not yet wired to the rendered forms — front-end submit handlers still need a small change to `POST /api/cms/submissions/...`.
- Inline `onclick="goto('page','sub')"` handlers inside the HTML strings still work — `ClientBootstrap` attaches a `goto` function to `window` that delegates to the Next.js router.
